# Contract Implementation Test Matrix

This matrix is the implementation handoff between the current Solidity interface ratchets and future contract code. It is intentionally local-only: it describes the unit/invariant tests the contract package must satisfy before any approved Orchard/testnet work.

## Local-only execution boundary

- No deployments, RPC calls, real wallets, private credentials, recovery material, or funds.
- Contract implementation tests must run against a local in-memory Hardhat network only until Clonners explicitly approves Orchard/testnet work.
- Use deterministic local accounts generated by the test runner only; never load external signer material in autonomous campaign runs.
- Keep the existing interface ratchet green before adding implementation files.
- Treat contract events as final proof truth; API/indexer state remains projection/cache.

## TradingVault test matrix

| ID | Behavior | Setup | Action | Expected assertions |
| --- | --- | --- | --- | --- |
| TV-01 | User deposit increases available balance | Local mock account owns test token balance and approves vault in local-only harness | User calls `deposit(token, amount)` | `Deposit` event emits user/token/amount; `balanceOf == availableBalanceOf`; `lockedBalanceOf == 0`; no operator-held balance can move funds. |
| TV-02 | User withdraws only caller-owned available balance | User has available vault balance and another user exists | Caller invokes `withdraw(token, amount)` | Only caller balance decreases; recipient is caller; `Withdraw` event emits caller/token/amount; arbitrary `user` withdrawal is impossible by surface and behavior. |
| TV-03 | Admin/operator cannot withdraw user funds | User deposits; admin/operator local account has role-like privileges if implementation introduces roles | Admin/operator attempts every exposed balance-moving user-withdrawal path | Calls revert or are absent; user available/locked balances and token ownership remain unchanged. |
| TV-04 | Locked balances cannot be withdrawn by the user until unlocked or settled | User deposit is locked through authorized settlement/order-manager hook | User calls `withdraw(token, lockedAmount)` | Withdraw reverts or is capped to available balance; locked balance stays locked until `unlockFromSettlement` or `settleLockedBalance`. |
| TV-05 | Settlement-only lock/unlock/move hooks reject non-settlement callers | User has available or locked balances; unauthorized local account exists | Unauthorized caller invokes `lockForSettlement`, `unlockFromSettlement`, or `settleLockedBalance` | Each call reverts; balances and lock accounting remain unchanged. Authorized settlement caller emits `BalanceLocked`, `BalanceUnlocked`, or `SettlementBalanceMoved` only for valid inputs. |
| TV-06 | Trading pause never becomes a broad withdrawal freeze | Implementation includes a pause/emergency control | Trading is paused and user has available balance | Order placement/settlement may pause, but normal withdrawal of available caller-owned balance remains possible unless a narrowly documented emergency withdrawal guard is added and separately approved. |

## Settlement test matrix

| ID | Behavior | Setup | Action | Expected assertions |
| --- | --- | --- | --- | --- |
| ST-01 | Valid signed fill settles once | Maker/taker local accounts sign orders; vault balances are available/locked; market enabled; fees under cap | Authorized relayer/test caller invokes `settle(fill, makerSignature, takerSignature)` | `TradeSettled` emits final event truth; maker/taker balances move exactly once; nonces are marked used; proof fields match `FillPacket`. |
| ST-02 | Fill rejects reused or cancelled nonces | Nonce is already used or cancelled in `NonceManager` | Submit otherwise valid fill | Settlement reverts before vault balance movement; no `TradeSettled` event; nonce state remains unchanged except existing cancellation/used marker. |
| ST-03 | Fill rejects expired orders and replay-domain mismatch | Fill has stale `expiresAt`, wrong `chainId`, or wrong `settlementContract` | Submit fill | Settlement reverts before balance movement and before nonce consumption; replay-domain fields are checked before proof event emission. |
| ST-04 | Fill rejects disabled markets and invalid price/amount constraints | Market disabled or fill violates signed price, amount, IOC/slippage, or minimum amount constraints | Submit fill | Settlement reverts; no vault movement; no proof event; matcher-side `ORDER_MATCHED` cannot become public fill truth. |
| ST-05 | Partial fill accounting cannot exceed signed order amounts | Existing cumulative filled amounts are near the signed cap | Submit a fill that would exceed maker or taker amount | Settlement reverts and leaves cumulative accounting unchanged; valid residual partial fill succeeds within signed amount bounds. |
| ST-06 | Fee cap and fee recipient are enforced before balance movement | Fill fees exceed `maxFeeBps` or fee recipient mismatches expected manager config | Submit fill | Settlement reverts before vault movement; valid fees emit maker/taker fee amounts and explicit fee recipient in `TradeSettled`. |
| ST-07 | TradeSettled event is the only public proof trigger | Matching engine emits `ORDER_MATCHED` but relayer has not confirmed settlement | Query proof/indexer projection before and after `TradeSettled` | Before event: private pending status only, no public proof. After event: proof row includes event identity, `fillId`, market, price, amount, fees, and explorer fields only when real chain evidence exists. |

## Dependency contract matrix

| Contract | Required implementation tests | Notes |
| --- | --- | --- |
| NonceManager | `NM-01`: `cancelNonce`, `cancelNonceRange`, `markNonceUsed` | Users can cancel their own nonces; settlement-only nonce consumption cannot be called by arbitrary accounts; used/cancelled nonces are replay-proof. |
| MarketRegistry | `MR-01`: enabled market metadata and precision/minimums; `MR-02`: Settlement dependency wiring | Markets start disabled until added; disabled/unknown/token-mismatched markets reject settlement before nonce/accounting/vault/proof mutation; precision/minimums are evented and stable for indexer proof replay. |
| FeeManager | `FM-01`: hard `maxFeeBps()` cap plus fee-update events | Fee changes emit deterministic events, cannot exceed cap, and should be timelock/multisig-gated before production. |
| DelegateKeyRegistry | `READ_ONLY`, trading/cancel permissions, `NO_WITHDRAW`, `NO_ADMIN` | Delegate keys can place/cancel within expiry, allowed markets, and max notional; they never gain withdrawal or admin authority in the MVP. |

## Approval gates before real Quai activity

- Adding a Hardhat test harness is allowed only for local/in-memory execution unless explicitly approved.
- Orchard deployments, contract verification, private keys, or transaction sends require explicit Clonners approval.
- Contract verification metadata and Quaiscan flows must be tested manually in an approved environment before any production-readiness claim.
- Real contract mode must reject proof projection unless it has settlement transaction hash, block number, block hash, event index, and explorer URL from event truth.

## Native Qi caveat

Native Qi is UTXO-model and must not be treated as an ERC-20 vault token until a wrapper/adapter/conversion design is proven. The mock `QI-QUAI` market may stay in API/UI demos, but real vault implementation tests must start with account-model assets that the contract can lock and settle safely.
